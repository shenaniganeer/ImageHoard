# Browser navigation, delete/archive wizard, and folder tree coordination

Agents and contributors: follow repository-wide guidance in [`AGENTS.md`](../../AGENTS.md) (source-of-truth order, PRD traceability, and closing `ImageHoard.App` before `dotnet build` / `dotnet test`).

This note describes how **image navigation**, **delete/move (wizard) processing**, and the **browser `TreeView`** interact. The subsystem spans several `MainWindow` partials; behavior is sensitive to **concurrent async** work.

## Maintenance rule

Any pull request that changes **keyboard/pointer dispatch for browse navigation**, **delete/archive wizard destructive flows**, **browser tree Find overlay**, **browser tree population, selection, or viewport scrolling** (`BrowserTreeViewportIntent`, `BrowserTreeViewportVisibility`, `BrowserTreeViewportPageScrollPlan`, `BrowserTreeViewportIntentResolver`, `BrowserPaneState`, `MainWindow.BrowserViewport.cs` / `BrowserTreeViewportPump`, `ScheduleViewport`, `ScheduleViewportAsync`, `RunColdBootViewportAsync`, `SuppressViewportForColdBoot`, `BuildBrowserPaneState`, `PrepareBrowserTreeViewportAfterWizardMutation`, `TryBringFolderTreeNodeIntoView`, **`paths.lastActedFsObject` / `SetLastActedFsObject` / `ForColdBootAnchor`**), **folder metrics queueing or UI apply for browser folder rows** (`StartFolderMetricsWorkAsync`, `ApplyFolderMetricsSnapshotCore`, `FolderMetricsSnapshotApplyKind`, trusted subtree cache vs immediate expand state, `FolderMetricsTrust`), **tree slideshow discovery or path-store behavior** (`TreeSlideshowSession`, `SlideshowDiscoveredPathStore`, `RecursiveImageEnumerator` slideshow enumeration, `SlideshowCoordinator`, `SlideshowAlgorithmDefaults`), **path overlay list position during slideshow** (`ApplyOverlayListPositionFromTreeAsync`, `SlideshowCoordinator.TryGetSlideshowOverlayListPosition`), **preview zoom / scroll metrics (`UpdatePreviewScrollMetrics`, `ApplyFullscreenImageForFitMode`, `TryGetPreviewImageIntrinsicDips`, `RequestViewZoomActualPixelsAsync` / `TryExecuteViewZoomActualPixels`, `ReloadPreviewAfterZoomChangeAsync`, cursor-relative zoom anchors: `TryPrepareZoomAnchorFromWheelIfNeeded`, `TryApplyPendingZoomScrollAnchorForHost`, `BeginZoomCommandScrollAnchorCapture`, `FullscreenScrollViewer` / `FullscreenScrollContentGrid`)**, or **preview enqueue/drain** in this area **must update this document** in the same PR (diagrams, flags, method names, or new entry points). Prefer linking to API symbols over pasting large code blocks.

## Browser tree selection and in-tree delete

- **`FolderTree`** uses default **single** `TreeView` selection (no `SelectionMode="Multiple"`). Pointer selection is handled by the control; **`FolderTree_SelectionChanged`** drives preview via **`ApplyBrowserTreeLeadPreviewAsync`** (image rows → **`OnFolderTreeImageRowSelectedAsync`**; otherwise **`ClearBrowserPaneImagePreviewStatePreserveTreeSelection`** when a preview was showing).
- **Context menu Delete** (`BrowserContextDelete_Click`, `MainWindow.BrowserTreeDelete.cs`): syncs selection to **`_browserContextMenuTargetNode`** when present, then **`ExecuteBrowserTreeDeleteForCurrentSelectionAsync`** (at most one nav row from **`FolderTree.SelectedNode`** via **`GetSelectedBrowserTreeNavNodes`**). After deletes, **`ClearBrowserTreeSelectionAfterDeleteAsync`** clears **`FolderTree.SelectedNode`** and refreshes preview.
- **Recycle Bin** deletes use **`WizardExecuteImageRecycleOrPermanentBatchAsync`** (files; **`ShellBulkRecycle`** / **`IFileOperation`** batch to Recycle Bin with sequential **`ShellRecycle`** fallback, then bounded-parallel **`File.Delete`** for user-approved permanent deletes) and **`ExecuteSendFolderToRecycleBinAfterConfirmAsync`** (folders) after **`ContentDialog`** confirmation; selected paths are filtered and ordered with **`BrowserTreeDeletePathDedupe`** (`ImageHoard.Core.Browse`).
- **Open archive wizard in this folder** (`BrowserContextOpenArchiveWizardInFolder_Click`, `_browserTreeContextMenu` in **`WireBrowserTreeTemplates`**, `MainWindow.BrowserPane.cs`): resolves the folder from the browse toolbar’s current folder or the context row (**`FolderTreeEntry.Path`** or **`ImageRow`**’s parent directory), **`SyncBrowseTreeSelection`** for tree rows, then **`ShowOrActivateDeleteArchiveWizardForFolder`**. The path must pass **`TreeDeletePathsAreUnderBrowseRoot`** (same containment rule as tree delete). The wizard stores **`_deleteArchiveWizardFolderPathOverride`** so **`TryGetDeleteArchiveWizardWorkingFolder`** returns that directory **before** **`TryResolveDeleteArchiveWizardDirectoryFromCurrentImage`**, enabling empty folders without a current image; the override is cleared in **`NotifyDeleteArchiveWizardClosed`** and at the start of **`ShowOrActivateDeleteArchiveWizard`** (keyboard **`sort.deleteArchiveWizard`**). Browse-root relocate (`MainWindow.BrowserPane.cs`) updates the override with **`RelocateFilesystemPathUnderRoot`** alongside **`_deleteArchiveWizardCapturedWorkingFolder`**.

While the delete/archive wizard is visible, **Escape** and **Close** are limited when `IsBrowserPaneMutationInProgress` is true: the overlay root handler and `TryDismissTopModalForEscape` consume Escape without dismissing; Close is disabled (`DeleteArchiveWizardPanel.SetBrowserPaneMutationBlocking`).

## Navigation pipeline (non-slideshow)

1. **`BrowseNavigateByStep`** (`MainWindow.BrowserPane.cs`) increments the nav counter, resets **`MouseButtonClickChainTracker`** (`_mouseButtonClickChainTracker.Reset()`), and starts **`BrowseNavigateByStepAsync`** without awaiting (fire-and-forget). **`OnPreviewImagePathCommitted`** (`MainWindow.PreviewScroll.cs`) also resets that tracker whenever the committed preview path changes or is cleared so a click after next/prev/first/last image (or slideshow advance, tree-driven preview, etc.) is **click 1**, not a spurious second click in the Win32 / **`PreviewImagePaneMultiClickThresholdMs`** window for chords such as **`Left` + `clickCount: 2`** bound to **`nav.lastImage`**.
2. **`BrowseNavigateByStepAsync`** resolves browse context from `_browseNavAnchorPath`, tree selection, `_currentImageFullPath`, `_currentFolderPath`, loads paths, computes index, then **`EnqueuePreviewNavigation`**, **`SetLastActedFsObject(nextPath)`**, and tree sync (**`SyncBrowseTreeSelection`** / **`TrySyncBrowseTreeSelectionToImagePathAsync`**), then **`ScheduleViewport(BrowserTreeViewportIntentResolver.ForImageStep(...))`**. The resulting intent uses **`BrowserTreeViewportVisibility.PageWhenOutsideViewport`**: the tree **`ScrollViewer`** does **not** re-pin the selected image row to the top on every step; if the row is fully visible, the viewport stays put, and if the row leaves the viewport the pump scrolls by **whole viewport heights** until the row fits (virtualized rows still rely on the existing **`LayoutUpdated`** retry loop until a **`TreeViewItem`** container exists). Other viewport reasons keep **`BringIntoView`** alignment behavior.
3. **`BrowseNavigateSiblingFolderFromInput`** → **`BrowseNavigateSiblingFolderAsync`** for `nav.nextDirectory` / `nav.prevDirectory` (and for **`browse.treeNext` / `browse.treePrevious`** when a folder row is selected). Optional **`contextDirectoryOverride`** skips **`BrowseContextDirectory.Resolve`** after validating the path is under the browse root (**`BrowseContextImageSequence.IsContextDirectoryUnderBrowseRoot`**). Uses **`_populateBrowserGeneration`** to drop stale results after repopulation; wizard **`RefreshBrowserPaneAfterWizardImageDeletesAsync` / `RefreshBrowserPaneAfterWizardUndoAsync`** and **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`** (via **`NavigateToFolderAsync`**) also advance this generation so staged folder population cannot interleave after tree reconciliation. When navigating among siblings under the same browse root (non-root context), after collapsing the prior folder the code **`await ScheduleViewportAsync(ForSiblingFolderNav(...))`** once so layout can settle inside the pump (no separate pre-collapse viewport pass). **`SetLastActedFsObject(targetFolderPath)`** on success.

When `IsBrowserPaneMutationInProgress` is true, **`BrowseNavigateByStep`** returns immediately (no async work); **`BrowseNavigateByStepAsync`** and **`BrowseNavigateSiblingFolderAsync`** no-op at entry.

## Preview drain

`MainWindow.PreviewNavigation.cs`: **`EnqueuePreviewNavigation`**, **`DrainPreviewNavigationCoreAsync`**, coalescing, **`SyncTreeSelectionToImagePath`**. Slideshow tree navigation is handled only through merged input chords (`TryExecuteInputCommand` → `TryMoveNextTree` / `TryMovePreviousTree`); when `IsBrowserPaneMutationInProgress` is true, those commands return handled without acting.

### Tree-scope slideshow overlay (list position)

When **`_slideshowUiActive`** and the overlay is in **tree** mode, the path overlay (`ApplyOverlayListPositionFromTreeAsync` in `MainWindow.BrowserPane.cs`) uses **`SlideshowCoordinator.TryGetSlideshowOverlayListPosition`** → **`TreeSlideshowSession.TryGetTreeOverlayPosition`**: text is **`N/M · discovered D`** where **`N`** is 1-based position in the **linear display history** (including redo), **`M`** is **`history.Count`**, and **`D`** is **`DiscoveredImageCount`** (paths appended by the background enumerator for this session). **Sibling** slideshow overlay continues to show ordered sibling **`index/total`** for that directory. See [`docs/design-decisions/slideshow-algorithm-p0.md`](../design-decisions/slideshow-algorithm-p0.md) for shuffle, uniform sampling, and RAM + spill storage.

**Preview user zoom** (`MainWindow.PreviewScroll.cs`): **`_previewUserZoomFactor`** scales the laid-out preview (and fullscreen) relative to the **Image fit** baseline computed from **EXIF-oriented file pixel dimensions** (`TryGetPreviewImageIntrinsicDips`; **1:1** fit mode still uses decoded bitmap dimensions because the decode may be max-edge capped). **`WicBitmapLoader.DecodeWithOrientationAsync`** returns a **`WicBitmapDecodeResult`** so each commit caches **`RememberPreviewOrientedPixelSize`** without a second file read. **`OnPreviewImagePathCommitted`** resets zoom to **1** when the **committed preview file path** changes (and clears pending zoom-scroll anchors); it also resets **`MouseButtonClickChainTracker`** on path change or clear so multi-click mouse chords do not carry across image navigation. **`ClearImageSelectionAndPreviewCore`** clears the anchor; **`NavigateToFolderAsync`** resets zoom on folder navigation; **`TryExecuteInputCommand`** handles **`view.zoomIn`**, **`view.zoomOut`**, **`view.zoomResetFit`**, and **`view.zoomActualPixels`** via **`TryRequestViewZoomActualPixelsFromInput`** → **`RequestViewZoomActualPixelsAsync`** → **`ViewZoomActualPixelsCoreAsync`** (Ctrl+wheel zoom is gated to the preview host / fullscreen scroll content via **`IsPointerChordAllowedForCommand`** for in/out only). **`view.zoomIn` / `view.zoomOut` / `view.zoomResetFit`** bump **`_previewInvalidateGeneration`** and **`ReloadPreviewAfterZoomChangeAsync`** (`MainWindow.PreviewNavigation.cs`) so **`CreateWicDecodeLayout`** (`MainWindow.DecodeTargets.cs`) can enlarge the WIC **Fit** target box by **`_previewUserZoomFactor`** and re-decode detail for the same path instead of compositor-upscaling a viewport-sized decode. **`view.zoomActualPixels`**: when **Shrink only** / **Shrink and stretch** have fit-decoded below the capped full-resolution size, **`WicBitmapLoader.GetOrientedPixelDimensionsAsync`** compares oriented file size to the current decode; if a larger WIC output is needed, **`SetForceNextPreviewFullResolutionDecode`** makes the next **`CreateWicDecodeLayout`** use **`BitmapUniformKind.FullResolution`** (same max-edge cap as **1:1** fit), then **`CommitPreviewImmediatelyAsync`** re-decodes; **`TryExecuteViewZoomActualPixels`** applies 1:1 layout using oriented intrinsics. **`RememberDecodeTargetBox`** records the **zoom-inclusive** physical Fit target after commits (and the viewport-only box after a full-resolution commit) so **`ReloadIfDecodeTargetBoxChangedAsync`** debounce compares consistently. Resize-only **`UpdatePreviewScrollMetrics`** on the same image does not reset zoom.

**Image fit modes** (`MainWindow.P0.cs` **`ImageFitMode`**, **`view.cycleFitMode`**): P0 exposes three radio modes under **Image → Image fit** plus **`view.zoomResetFit`**. **`Shrink only`** and **`Shrink & stretch`** both decode with **`BitmapUniformKind.Fit`** against the preview/fullscreen target box multiplied by **`_previewUserZoomFactor`** (`CreateWicDecodeLayout` in `MainWindow.DecodeTargets.cs`). Display baseline (`ComputeImageDisplayBaselineDipsWindowed` / `ComputeImageDisplayBaselineDipsFullscreen` in `MainWindow.PreviewScroll.cs`): **`Shrink & stretch`** uses uniform **contain** scale via **`PreviewCoverLayout.UniformContainScale`** (entire image inside the viewport, including compositor upscale when the laid-out intrinsic is smaller than the viewport). **`Shrink only`** uses **`PreviewCoverLayout.UniformContainScaleShrinkOnly`** (same contain rule capped at oriented intrinsic logical size, so no upscaling at zoom 1). **`1:1`** uses full-resolution decode and decoded DIP baseline. **`ApplyFullscreenImageForFitMode`** applies the same baseline × zoom sizing for the fullscreen `Image` inside **`FullscreenScrollContentGrid`** (hosted by **`FullscreenScrollViewer`**, mirroring the windowed preview scroll host).

**Cursor-relative zoom (preview + fullscreen)** (`MainWindow.PreviewScroll.cs`, `MainWindow.xaml.cs`): Before **`_previewUserZoomFactor`** changes, the app records a pending anchor (viewport point in DIP relative to the active **`ScrollViewer`**, plus normalized **(u,v)** on the displayed image rect) from the **mouse wheel** when **`TryPrepareZoomAnchorFromWheelIfNeeded`** runs ahead of **`TryExecuteInputCommand`** for **`view.zoomIn`**, **`view.zoomOut`**, or **`view.zoomResetFit`** (chord allowed only over **`PreviewHostGrid`** or **`FullscreenScrollContentGrid`**). Keyboard/toolbar zoom calls **`BeginZoomCommandScrollAnchorCapture`**, which anchors to the **viewport center** unless the wheel path set **`_suppressNextZoomAnchorCenterCapture`**. After layout, **`TryApplyPendingZoomScrollAnchorForHost`** adjusts **`ScrollViewer.ChangeView`** so the same image pixel stays under that viewport point (clamped to scroll extents). **`PrepareFullscreenEnterNavigation`** / **`PrepareFullscreenExitNavigation`** reset scroll and clear pending anchors around fullscreen toggles. **`view.panPreview`** drags update whichever **`ScrollViewer`** is active (preview vs fullscreen).

### Sort-flag target path vs preview (`TryGetSortFlagTargetPath`)

`MainWindow.BrowserPane.cs`: **`TryGetSortFlagTargetPath`** chooses the file that **Keep / Delete / Unset** commands (`sort.flag*` and legacy K/D/U) affect. The info overlay’s flag glyph is driven by **`_currentImageFullPath`**, so if the tree’s selected **`ImageRow`** lags the committed preview (e.g. slideshow navigation without a matching selection update), commands must still target the **preview** path or the UI looks broken in fullscreen.

Resolution order:

1. While **`_slideshowUiActive`** and the preview path is non-empty → use **`_currentImageFullPath`**.
2. Else if the tree has an **`ImageRow`** selected and it **differs** from **`_currentImageFullPath`** (both set) → use **`_currentImageFullPath`** (preview wins over stale selection).
3. Else if an **`ImageRow`** is selected → use that row’s path (including “selection with no preview” cases).
4. Else → **`_currentImageFullPath`** when set.

After a successful bitmap commit in **`DecodeAndCommitPreviewAsync`**, when **`_slideshowUiActive`**, the app calls **`SyncTreeSelectionToImagePath(path)`** (suppresses selection-changed preview enqueue so **`DiscardSlideshowSession`** is not run) so the tree anchor stays aligned with the slide for other features.

## Destructive / reconciliation pipeline (wizard + browser)

| Step | API | Role |
|------|-----|------|
| User confirms delete/move | `WizardExecute*` in `MainWindow.DeleteArchiveWizard.cs` | File system and navigation side effects. **Move to archive** builds a case-insensitive union of (optional) inverse-keep paths and all **Delete**-flagged image paths in the folder, then runs **`WizardExecuteImageRecycleOrPermanentBatchAsync`** (parallel pre-scan; shell batch recycle via **`ShellBulkRecycle`** with **`ShellRecycle`** fallback; user-approved permanent deletes use bounded parallelism) before `MergeMoveDirectoryAsync`. When that batch is immediately followed by a directory merge, the batch may pass **`deferBrowserPaneRefresh: true`** so the UI reconciles once after the merge succeeds (or once on merge failure using captured delete stats). |
| Reconcile tree + preview | `RefreshBrowserPaneAfterWizardImageDeletesAsync`, `RefreshBrowserPaneAfterWizardUndoAsync` (`MainWindow.BrowserPane.cs`) | Syncs image rows with disk, updates aggregates, refocuses selection. Wrapped with **`EnterBrowserPaneMutation` / `LeaveBrowserPaneMutation`** (try/finally). Each method **`Interlocked.Increment`s `_populateBrowserGeneration`** at the start of the refresh so in-flight **`AppendBrowserStagedToTargetAsync`** work from the prior browse cannot resume with a stale token after disk sync. |
| Wizard parent navigation (single bump + viewport) | **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`** (`MainWindow.BrowserPane.cs`) | After **folder recycle** or **Move to archive** when the removed folder was the browsed folder: calls **`NavigateToFolderAsync(parent, suppressViewportAfterRootPopulate: true)`** (one generation increment and root repopulate **without** populate-time viewport scheduling), then **`ClearImageSelectionAndPreviewCore`**, **`CommitIncrementalFolderPreviewAndSelectionAsync`**, and **one** awaited **`ScheduleViewportAsync(BrowserTreeViewportIntentResolver.ForWizardCommit(..., BrowserTreeViewportReason.AfterWizardNavigateToParent, ...))`** (the pump’s first step still runs **`PrepareBrowserTreeViewportAfterWizardMutation`** so deferred sibling resorts flush before scroll). | Cold boot session restore | **`RestoreStartupBrowseAsync`** (`MainWindow.xaml.cs`), **`NavigateToFolderAsync(..., coldBootSessionRestore: true)`**, **`PopulateBrowserRootsCoreAsync(..., deferRootSelectionForColdBootRestore)`**, **`FinalizeBrowserRootPopulateAfterImmediateAppend`** / **`FinalizeBrowserRootPopulateSelectionAndChrome`**, **`RestoreColdBootBrowseAfterNavigateAsync`** (`MainWindow.BrowserPane.cs`), **`SuppressViewportForColdBoot`** / **`_suppressBrowserTreeViewportMutationForColdBoot`**, **`RunColdBootViewportAsync`**, **`BrowserTreeViewportIntentResolver.ForColdBootAnchor`**, **`paths.lastActedFsObject`** (`AppSettingsStore` / `AppSessionSettings`) | See **Cold boot and tree scroll persistence** below. First-time expand still uses **`PopulateFolderTreeNodeChildrenAsync`**. |

Wizard `Refresh*` stops that timer and applies any **pending** coalesced sibling sorts **after** commit and **before** **`ScheduleViewportAsync`** so the tree order matches sort keys when pinning. No-op for name / date sorts (no deferred pending set). |

### Browser tree viewport (intent + pump)

**Intent (Core)** — `BrowserTreeViewportIntent` is a small immutable record: **`Reason`**, **`PrimaryPath`** (folder or file row target), **`SecondaryPath`** (fallback row, e.g. parent of a file hit), **`VerticalAlignmentRatio`** (0 = pin row to top / leading; 0.5 = vertically center — cold-boot file anchor), **`PreferSelectionFirst`**, and **`Visibility`** (`BrowserTreeViewportVisibility`: default **`BringIntoView`**; **`ForImageStep`** alone sets **`PageWhenOutsideViewport`**). Reasons include cold boot, wizard delete/undo/parent-nav, folder navigation, sibling folder navigation, Find folder/file hit, image-step navigation, keyboard tree move, and root populate (`BrowserTreeViewportReason`).

**Resolver (Core)** — `BrowserTreeViewportIntentResolver` maps **`BrowserPaneState`** (+ optional wizard refocus context, find match, target paths) into an intent. Pin-after-commit rules mirror the former `GetBrowserTreeViewportPinPathAfterBrowseCommit` / browsed-directory resolution (still implemented in the host where needed; resolver exposes **`GetPinPathAfterBrowseCommit`** for tests and parity). Cold boot uses **`ForColdBootAnchor(state, lastActedFsObject)`** (null/invalid/outside-root → scroll list top; folder → **`PrimaryPath`** + ratio 0; existing file → file **`PrimaryPath`**, parent **`SecondaryPath`**, ratio 0.5, **`PreferSelectionFirst: true`**, **`Visibility: BringIntoView`** — unchanged center/top pinning vs session restore).

**Pump (App)** — `BrowserTreeViewportPump` (nested in `MainWindow.BrowserViewport.cs`) owns timing:

- **Coalescing** — Multiple **`ScheduleViewportAsync`** calls before the pump drain runs collapse to the **latest** intent; every waiter **`Task` completes** when that drain batch finishes. Intents with **`Reason == ColdBootRestore`** are **not** coalesced in the normal drain: **`ScheduleViewportAsync`** forwards them immediately to **`RunColdBootViewportAsync`** (restore code also invokes **`RunColdBootViewportAsync`** directly after selection sync).
- **Suppression** — While **`SuppressViewportForColdBoot(true)`** / **`_suppressBrowserTreeViewportMutationForColdBoot`** is set, **`ScheduleViewportAsync`** / **`ScheduleViewport`** return immediately without work (incidental viewport during restore is dropped).
- **Scroll pass** — Each run calls **`PrepareBrowserTreeViewportAfterWizardMutation`** first (flush deferred aggregate/count sibling resorts so order is final). Then **`TryScrollPassOnce`**: **`FolderTree.UpdateLayout()`**. When **`intent.Visibility == PageWhenOutsideViewport`** (image-step only), **`TryScrollPassPageWhenOutsideViewportOnce`** resolves the image **`TreeViewNode`** from **`PrimaryPath`**, measures the realized **`TreeViewItem`** against the inner folder-tree **`ScrollViewer`**, and applies **`BrowserTreeViewportPageScrollPlan.Compute`** → **`ScrollViewer.ChangeView`** in **viewport-height** steps when the row is not fully visible; it does **not** call **`TryBringFolderTreeNodeIntoView`** (so stepping does not continually align the row to the leading edge). Otherwise: resolve nodes from **`PrimaryPath`** / **`SecondaryPath`** (folders via path index; files via image-row lookup), **`PreferSelectionFirst`** vs pin-first ordering, and **`TryBringFolderTreeNodeIntoView(node, intent.VerticalAlignmentRatio)`** (inner **`TreeViewList.ScrollIntoView`** uses **`Leading`** when ratio ≤ 0, else **`Default`**; **`TreeViewItem.StartBringIntoView`** passes **`VerticalAlignmentRatio`**). Cold boot with no primary/secondary paths calls **`TryScrollFolderTreeToTop`**. If the target is not realized, the pump waits on **`FolderTree.LayoutUpdated`** (capped by **`ViewportPumpMaxWaitMs`** (250 ms) and **`ViewportPumpMaxLayoutCycles`** (8)), then retries. **Dispatcher priority**: drain starts at **Normal** when either path field is set (pinned-ish intents), else **Low**.
- **Exhaustion** — After the wait loop, a final **Normal**-priority dispatcher callback runs a best-effort fallback. For **`PageWhenOutsideViewport`**, it retries only the page-scroll path (no alignment fallback to avoid re-pinning the selection to the top). Otherwise the legacy fallback applies (including **`TryScrollFolderTreeToTop`** only when there is no valid browse directory **and** the intent is not “pinned” to an existing folder primary path — avoids snapping away from a meaningful browse pin on timeout).

**Cold boot inside the pump** — **`RunColdBootViewportAsync`**: for **`ColdBootRestore`**, flush resorts via **`PrepareBrowserTreeViewportAfterWizardMutation`**, **`UpdateLayout`**, wait one layout slice, then **`ExecuteScrollPassWithLayoutWaitAsync`** once (no persisted **`ScrollViewer`** offset restore; anchor alignment only).

### Cold boot and tree scroll persistence

1. **`RestoreStartupBrowseAsync`** sets **`SuppressViewportForColdBoot(true)`** before **`NavigateToFolderAsync(..., suppressViewportAfterRootPopulate: true, coldBootSessionRestore: true)`** and **`RestoreColdBootBrowseAfterNavigateAsync`**, then clears suppression in **`finally`**. While the flag is set, **`ScheduleViewport` / `ScheduleViewportAsync`** return without running the pump (incidental viewport work during restore is dropped).
2. **`coldBootSessionRestore`** defers root image selection (`deferRootSelectionForColdBootRestore`) so **`LastSelectedImage`** is not overwritten by **`FirstImageNodePreorder`** before subtrees exist.
3. **`RestoreColdBootBrowseAfterNavigateAsync`** — If the persisted **`BrowserTreeSessionSnapshot`** does not match the current browse root, resets the tree **`ScrollViewer`** to origin; otherwise expands persisted **`ExpandedFolderPaths`** shallow-first via **`TryEnsureFolderPathMaterializedAsync(..., expandAndPopulateDestinationFolder: true)`**. Then, when **`LastActedFsObject`** is set and under the browse root: **folder** anchors expand their ancestor chain plus the folder, **`SyncBrowseTreeSelection`** to the folder row, **`ApplyBrowserTreeLeadPreviewAsync`**; **file** anchors expand parent folders, **`TrySyncBrowseTreeSelectionToImagePathAsync`**, **`ApplyBrowserTreeLeadPreviewAsync`**. Finally **`await RunColdBootViewportAsync(BrowserTreeViewportIntentResolver.ForColdBootAnchor(BuildBrowserPaneState(), LastActedFsObject))`** — folder rows pin to the **top** of the viewport (`VerticalAlignmentRatio` 0); file rows use **center** alignment (ratio 0.5). Missing/stale/outside-root anchors yield a **top-of-list** scroll (empty primary/secondary in the intent). Cold-boot intents use default **`BrowserTreeViewportVisibility.BringIntoView`** (not **`PageWhenOutsideViewport`**).
4. **`FolderTreeScrollViewer_ViewChanged`** (`MainWindow.BrowserPane.cs`) skips **`SchedulePersistLayoutDebounced`** while **`_suppressBrowserTreeViewportMutationForColdBoot`** is true (non-intermediate events only), so the first restored **`ChangeView`** is not immediately overwritten in session state by layout churn.
5. **`CaptureBrowserTreeSnapshotIntoSessionIfBrowsing`** persists **`ExpandedFolderPaths`** only (no tree **`ScrollViewer`** offsets). **`paths.lastActedFsObject`** is updated by **`SetLastActedFsObject`** on user actions (see maintenance table row **Cold-boot anchor**).

## Races addressed

1. **Overlay dismissed while work continues** — Escape/Close could hide the wizard while `async` work still ran; `IsDeleteArchiveWizardOverlayOpen` no longer blocked `nav.*`. Mutation depth blocks browse commands and async browse entry points until teardown finishes.
2. **Overlapping `BrowseNavigateByStepAsync` instances** — Still possible when depth is zero; with depth, new navigations are suppressed during mutation.
3. **Viewport vs navigation** — `RefreshBrowserPaneAfterWizard*` awaits **`ScheduleViewportAsync(...)`** (viewport pump) before clearing nested mutation depth so the tree scroll pass finishes (within the implemented completion contract) before navigation unblocks. **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`** uses the same awaited **`ScheduleViewportAsync(ForWizardCommit(...))`** after a single populate + commit when the browsed folder was removed.
4. **Viewport vs deferred folder resort** — Without a pre-viewport flush, the awaited scroll could run while sibling folder rows were still in pre-resort order, then a timer-driven **`FlushCoalescedFolderResorts`** would reorder and rely on a second **`ScheduleViewport`**. Wizard `Refresh*` still calls **`PrepareBrowserTreeViewportAfterWizardMutation`** immediately before **`ScheduleViewportAsync`**; the pump repeats that flush at the start of each scroll pass so ordering stays aligned with the awaited pass and with later drains.
5. **Staged browse populate vs wizard tree refresh** — `AppendBrowserStagedToTargetAsync` used to append all image `TreeViewNode`s before the first `_populateBrowserGeneration` check, then insert folder chunks across `await Task.Yield()` boundaries while wizard `Refresh*` could reconcile the same sibling list under the same generation token. **`RefreshBrowserPaneAfterWizard*`** now increments **`_populateBrowserGeneration`** at entry, and staged append validates the token **before** appending image rows so superseded work cannot leave partial lists.
6. **Cold boot vs deferred folder resort** — **`FlushCoalescedFolderResorts`** ends with **`ScheduleViewport(ForFolderNavigation(...))`**. While **`_suppressBrowserTreeViewportMutationForColdBoot`** is true, that **`ScheduleViewport`** call is skipped so cold-boot restore is not disturbed. **`SuppressViewportForColdBoot`** still keeps **`ScheduleViewport` / `ScheduleViewportAsync`** as no-ops for the whole **`RestoreStartupBrowseAsync`** try block. **`FolderTreeScrollViewer_ViewChanged`** skips **`SchedulePersistLayoutDebounced`** while suppression is active.

## Cold boot manual QA (browser tree)

After changing cold-boot restore, verify on a full app restart (process exit, not just navigate away):

1. **Deep folder expand** — Expand a nested subfolder (no image preview required), exit, restart: that folder row is **at the top** of the tree viewport and **expanded**.
2. **Find folder hit** — Find-in-tree selects a deep folder match, exit, restart: same (**top**, expanded chain).
3. **Find file hit** — Find selects an image row, restart: ancestors expanded, **image row vertically centered** in the tree viewport.
4. **Image step / last image** — **`LastActedFsObject`** / legacy **`lastSelectedImage`** seed: after stepping images, restart: current file row **centered**; preview matches.
5. **No anchor** — Clear or move off-disk anchor, or first launch with no `lastActedFsObject`: tree list scrolls to **top**.
6. **Different browse root** — Open another browse folder (anchor cleared), restart: tree at **top** under the new root until a new tree action.
7. **Aggregate folder sort** — Folder list sort **Total size** or **Image file count** so metrics resort runs; cold boot still lands on the anchored row (no spurious **`FlushCoalescedFolderResorts`** viewport during suppression).
