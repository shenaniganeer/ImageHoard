# Browser navigation, delete/archive wizard, and folder tree coordination

Agents and contributors: follow repository-wide guidance in [`AGENTS.md`](../../AGENTS.md) (source-of-truth order, PRD traceability, and closing `ImageHoard.App` before `dotnet build` / `dotnet test`).

This note describes how **image navigation**, **delete/move (wizard) processing**, and the **browser `TreeView`** interact. The subsystem spans several `MainWindow` partials; behavior is sensitive to **concurrent async** work.

## Maintenance rule

Any pull request that changes **keyboard/pointer dispatch for browse navigation**, **delete/archive wizard destructive flows**, **browser tree Find overlay**, **browser tree population, selection, or viewport scrolling**, **folder metrics queueing or UI apply for browser folder rows** (`StartFolderMetricsWorkAsync`, `ApplyFolderMetricsSnapshotCore`, `FolderMetricsSnapshotApplyKind`, trusted subtree cache vs immediate expand state, `FolderMetricsTrust`), **tree slideshow discovery or path-store behavior** (`TreeSlideshowSession`, `SlideshowDiscoveredPathStore`, `RecursiveImageEnumerator` slideshow enumeration, `SlideshowCoordinator`, `SlideshowAlgorithmDefaults`), **path overlay list position during slideshow** (`ApplyOverlayListPositionFromTreeAsync`, `SlideshowCoordinator.TryGetSlideshowOverlayListPosition`), **preview zoom / scroll metrics (`UpdatePreviewScrollMetrics`, `ApplyFullscreenImageForFitMode`, `TryGetPreviewImageIntrinsicDips`, `RequestViewZoomActualPixelsAsync` / `TryExecuteViewZoomActualPixels`, `ReloadPreviewAfterZoomChangeAsync`, cursor-relative zoom anchors: `TryPrepareZoomAnchorFromWheelIfNeeded`, `TryApplyPendingZoomScrollAnchorForHost`, `BeginZoomCommandScrollAnchorCapture`, `FullscreenScrollViewer` / `FullscreenScrollContentGrid`)**, or **preview enqueue/drain** in this area **must update this document** in the same PR (diagrams, flags, method names, or new entry points). Prefer linking to API symbols over pasting large code blocks.

## Browser tree selection and in-tree delete

- **`FolderTree`** uses default **single** `TreeView` selection (no `SelectionMode="Multiple"`). Pointer selection is handled by the control; **`FolderTree_SelectionChanged`** drives preview via **`ApplyBrowserTreeLeadPreviewAsync`** (image rows → **`OnFolderTreeImageRowSelectedAsync`**; otherwise **`ClearBrowserPaneImagePreviewStatePreserveTreeSelection`** when a preview was showing).
- **Context menu Delete** (`BrowserContextDelete_Click`, `MainWindow.BrowserTreeDelete.cs`): syncs selection to **`_browserContextMenuTargetNode`** when present, then **`ExecuteBrowserTreeDeleteForCurrentSelectionAsync`** (at most one nav row from **`FolderTree.SelectedNode`** via **`GetSelectedBrowserTreeNavNodes`**). After deletes, **`ClearBrowserTreeSelectionAfterDeleteAsync`** clears **`FolderTree.SelectedNode`** and refreshes preview.
- **Recycle Bin** deletes use **`WizardExecuteImageRecycleOrPermanentBatchAsync`** (files; **`ShellBulkRecycle`** / **`IFileOperation`** batch to Recycle Bin with sequential **`ShellRecycle`** fallback, then bounded-parallel **`File.Delete`** for user-approved permanent deletes) and **`ExecuteSendFolderToRecycleBinAfterConfirmAsync`** (folders) after **`ContentDialog`** confirmation; selected paths are filtered and ordered with **`BrowserTreeDeletePathDedupe`** (`ImageHoard.Core.Browse`).
- **Open archive wizard in this folder** (`BrowserContextOpenArchiveWizardInFolder_Click`, `_browserTreeContextMenu` in **`WireBrowserTreeTemplates`**, `MainWindow.BrowserPane.cs`): resolves the folder from the browse toolbar’s current folder or the context row (**`FolderTreeEntry.Path`** or **`ImageRow`**’s parent directory), **`SyncBrowseTreeSelection`** for tree rows, then **`ShowOrActivateDeleteArchiveWizardForFolder`**. The path must pass **`TreeDeletePathsAreUnderBrowseRoot`** (same containment rule as tree delete). The wizard stores **`_deleteArchiveWizardFolderPathOverride`** so **`TryGetDeleteArchiveWizardWorkingFolder`** returns that directory **before** **`TryResolveDeleteArchiveWizardDirectoryFromCurrentImage`**, enabling empty folders without a current image; the override is cleared in **`NotifyDeleteArchiveWizardClosed`** and at the start of **`ShowOrActivateDeleteArchiveWizard`** (keyboard **`sort.deleteArchiveWizard`**). Browse-root relocate (`MainWindow.BrowserPane.cs`) updates the override with **`RelocateFilesystemPathUnderRoot`** alongside **`_deleteArchiveWizardCapturedWorkingFolder`**.

While the delete/archive wizard is visible, **Escape** and **Close** are limited when `IsBrowserPaneMutationInProgress` is true: the overlay root handler and `TryDismissTopModalForEscape` consume Escape without dismissing; Close is disabled (`DeleteArchiveWizardPanel.SetBrowserPaneMutationBlocking`).

## Navigation pipeline (non-slideshow)

1. **`BrowseNavigateByStep`** (`MainWindow.BrowserPane.cs`) increments the nav counter and starts **`BrowseNavigateByStepAsync`** without awaiting (fire-and-forget).
2. **`BrowseNavigateByStepAsync`** resolves browse context from `_browseNavAnchorPath`, tree selection, `_currentImageFullPath`, `_currentFolderPath`, loads paths, computes index, then **`EnqueuePreviewNavigation`** and tree sync.
3. **`BrowseNavigateSiblingFolderFromInput`** → **`BrowseNavigateSiblingFolderAsync`** for `nav.nextDirectory` / `nav.prevDirectory` (and for **`browse.treeNext` / `browse.treePrevious`** when a folder row is selected). Optional **`contextDirectoryOverride`** skips **`BrowseContextDirectory.Resolve`** after validating the path is under the browse root (**`BrowseContextImageSequence.IsContextDirectoryUnderBrowseRoot`**). Uses **`_populateBrowserGeneration`** to drop stale results after repopulation; wizard **`RefreshBrowserPaneAfterWizardImageDeletesAsync` / `RefreshBrowserPaneAfterWizardUndoAsync`** and **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`** (via **`NavigateToFolderAsync`**) also advance this generation so staged folder population cannot interleave after tree reconciliation.

When `IsBrowserPaneMutationInProgress` is true, **`BrowseNavigateByStep`** returns immediately (no async work); **`BrowseNavigateByStepAsync`** and **`BrowseNavigateSiblingFolderAsync`** no-op at entry.

## Preview drain

`MainWindow.PreviewNavigation.cs`: **`EnqueuePreviewNavigation`**, **`DrainPreviewNavigationCoreAsync`**, coalescing, **`SyncTreeSelectionToImagePath`**. Slideshow tree navigation is handled only through merged input chords (`TryExecuteInputCommand` → `TryMoveNextTree` / `TryMovePreviousTree`); when `IsBrowserPaneMutationInProgress` is true, those commands return handled without acting.

### Tree-scope slideshow overlay (list position)

When **`_slideshowUiActive`** and the overlay is in **tree** mode, the path overlay (`ApplyOverlayListPositionFromTreeAsync` in `MainWindow.BrowserPane.cs`) uses **`SlideshowCoordinator.TryGetSlideshowOverlayListPosition`** → **`TreeSlideshowSession.TryGetTreeOverlayPosition`**: text is **`N/M · discovered D`** where **`N`** is 1-based position in the **linear display history** (including redo), **`M`** is **`history.Count`**, and **`D`** is **`DiscoveredImageCount`** (paths appended by the background enumerator for this session). **Sibling** slideshow overlay continues to show ordered sibling **`index/total`** for that directory. See [`docs/design-decisions/slideshow-algorithm-p0.md`](../design-decisions/slideshow-algorithm-p0.md) for shuffle, uniform sampling, and RAM + spill storage.

**Preview user zoom** (`MainWindow.PreviewScroll.cs`): **`_previewUserZoomFactor`** scales the laid-out preview (and fullscreen) relative to the **Image fit** baseline computed from **EXIF-oriented file pixel dimensions** (`TryGetPreviewImageIntrinsicDips`; **1:1** fit mode still uses decoded bitmap dimensions because the decode may be max-edge capped). **`WicBitmapLoader.DecodeWithOrientationAsync`** returns a **`WicBitmapDecodeResult`** so each commit caches **`RememberPreviewOrientedPixelSize`** without a second file read. **`OnPreviewImagePathCommitted`** resets zoom to **1** when the **committed preview file path** changes (and clears pending zoom-scroll anchors); **`ClearImageSelectionAndPreviewCore`** clears the anchor; **`NavigateToFolderAsync`** resets zoom on folder navigation; **`TryExecuteInputCommand`** handles **`view.zoomIn`**, **`view.zoomOut`**, **`view.zoomResetFit`**, and **`view.zoomActualPixels`** via **`TryRequestViewZoomActualPixelsFromInput`** → **`RequestViewZoomActualPixelsAsync`** → **`ViewZoomActualPixelsCoreAsync`** (Ctrl+wheel zoom is gated to the preview host / fullscreen scroll content via **`IsPointerChordAllowedForCommand`** for in/out only). **`view.zoomIn` / `view.zoomOut` / `view.zoomResetFit`** bump **`_previewInvalidateGeneration`** and **`ReloadPreviewAfterZoomChangeAsync`** (`MainWindow.PreviewNavigation.cs`) so **`CreateWicDecodeLayout`** (`MainWindow.DecodeTargets.cs`) can enlarge the WIC **Fit** target box by **`_previewUserZoomFactor`** and re-decode detail for the same path instead of compositor-upscaling a viewport-sized decode. **`view.zoomActualPixels`**: when **Shrink only** / **Shrink and stretch** have fit-decoded below the capped full-resolution size, **`WicBitmapLoader.GetOrientedPixelDimensionsAsync`** compares oriented file size to the current decode; if a larger WIC output is needed, **`SetForceNextPreviewFullResolutionDecode`** makes the next **`CreateWicDecodeLayout`** use **`BitmapUniformKind.FullResolution`** (same max-edge cap as **1:1** fit), then **`CommitPreviewImmediatelyAsync`** re-decodes; **`TryExecuteViewZoomActualPixels`** applies 1:1 layout using oriented intrinsics. **`RememberDecodeTargetBox`** records the **zoom-inclusive** physical Fit target after commits (and the viewport-only box after a full-resolution commit) so **`ReloadIfDecodeTargetBoxChangedAsync`** debounce compares consistently. Resize-only **`UpdatePreviewScrollMetrics`** on the same image does not reset zoom.

**Image fit modes** (`MainWindow.P0.cs` **`ImageFitMode`**, **`view.cycleFitMode`**): P0 exposes three radio modes under **Image → Image fit** plus **`view.zoomResetFit`**. **`Shrink only`** and **`Shrink & stretch`** both decode with **`BitmapUniformKind.Fit`** against the preview/fullscreen target box multiplied by **`_previewUserZoomFactor`** (`CreateWicDecodeLayout` in `MainWindow.DecodeTargets.cs`). Display baseline (`ComputeImageDisplayBaselineDipsWindowed` / `ComputeImageDisplayBaselineDipsFullscreen` in `MainWindow.PreviewScroll.cs`): **`Shrink & stretch`** uses uniform **contain** scale via **`PreviewCoverLayout.UniformContainScale`** (entire image inside the viewport, including compositor upscale when the laid-out intrinsic is smaller than the viewport). **`Shrink only`** uses **`PreviewCoverLayout.UniformContainScaleShrinkOnly`** (same contain rule capped at oriented intrinsic logical size, so no upscaling at zoom 1). **`1:1`** uses full-resolution decode and decoded DIP baseline. **`ApplyFullscreenImageForFitMode`** applies the same baseline × zoom sizing for the fullscreen `Image` inside **`FullscreenScrollContentGrid`** (hosted by **`FullscreenScrollViewer`**, mirroring the windowed preview scroll host).

**Cursor-relative zoom (preview + fullscreen)** (`MainWindow.PreviewScroll.cs`, `MainWindow.xaml.cs`): Before **`_previewUserZoomFactor`** changes, the app records a pending anchor (viewport point in DIP relative to the active **`ScrollViewer`**, plus normalized **(u,v)** on the displayed image rect) from the **mouse wheel** when **`TryPrepareZoomAnchorFromWheelIfNeeded`** runs ahead of **`TryExecuteInputCommand`** for **`view.zoomIn`**, **`view.zoomOut`**, or **`view.zoomResetFit`** (chord allowed only over **`PreviewHostGrid`** or **`FullscreenScrollContentGrid`**). Keyboard/toolbar zoom calls **`BeginZoomCommandScrollAnchorCapture`**, which anchors to the **viewport center** unless the wheel path set **`_suppressNextZoomAnchorCenterCapture`**. After layout, **`TryApplyPendingZoomScrollAnchorForHost`** adjusts **`ScrollViewer.ChangeView`** so the same image pixel stays under that viewport point (clamped to scroll extents). **`PrepareFullscreenEnterNavigation`** / **`PrepareFullscreenExitNavigation`** reset scroll and clear pending anchors around fullscreen toggles. **`view.panPreview`** drags update whichever **`ScrollViewer`** is active (preview vs fullscreen).

### Sort-flag target path vs preview (`TryGetSortFlagTargetPath`)

`MainWindow.BrowserPane.cs`: **`TryGetSortFlagTargetPath`** chooses the file that **Keep / Delete / Unset** commands (`sort.flag*` and legacy K/D/U) affect. The info overlay’s flag glyph is driven by **`_currentImageFullPath`**, so if the tree’s selected **`ImageRow`** lags the committed preview (e.g. slideshow navigation without a matching selection update), commands must still target the **preview** path or the UI looks broken in fullscreen.

Resolution order:

1. While **`_slideshowUiActive`** and the preview path is non-empty → use **`_currentImageFullPath`**.
2. Else if the tree has an **`ImageRow`** selected and it **differs** from **`_currentImageFullPath`** (both set) → use **`_currentImageFullPath`** (preview wins over stale selection).
3. Else if an **`ImageRow`** is selected → use that row’s path (including “selection with no preview” cases).
4. Else → **`_currentImageFullPath`** when set.

After a successful bitmap commit in **`DecodeAndCommitPreviewAsync`**, when **`_slideshowUiActive`**, the app calls **`SyncTreeSelectionToImagePath(path)`** (suppresses selection-changed preview enqueue so **`DiscardSlideshowSession`** is not run) so the tree anchor stays aligned with the slide for other features.

Wizard `Refresh*` stops that timer and applies any **pending** coalesced sibling sorts **after** commit and **before** the awaited viewport so the tree order matches sort keys when pinning. No-op for name / date sorts (no deferred pending set). |
| Viewport | `ScheduleBrowserTreeViewportAfterMutation` (fire-and-forget) and **`ScheduleBrowserTreeViewportAfterMutationAsync`** (awaitable) | Pins the browsed folder row (or selection) into view after mutations; retries on `DispatcherQueue` when containers are not realized. **`TryBringFolderTreeNodeToTop`** (`MainWindow.BrowserPane.cs`) walks the `FolderTree` visual tree for the inner **`TreeViewList`** and calls **`ScrollIntoView(node, Leading)`** before **`TreeViewItem.StartBringIntoView`** when the row’s **`TreeViewItem`** is realized; it returns **`false`** until a container exists so **`ExecuteBrowserTreeViewportAfterMutationCoreAsync`** can retry instead of treating **`ScrollIntoView`** alone as success. Coalesced drain start and pinned-path **`TryStep`** retries use **Normal** priority when a pin path is supplied (unpinned stays **Low**). Sibling-folder navigation (**`BrowseNavigateSiblingFolderAsync`**) awaits an awaited viewport pass **before** collapsing the prior context folder and **again** after collapse so flatten/layout does not leave the **`ScrollViewer`** at offset zero. Wizard **`RefreshBrowserPaneAfterWizardImageDeletesAsync`** / **`RefreshBrowserPaneAfterWizardUndoAsync`** pass a pin path from **`GetBrowserTreeViewportPinPathAfterBrowseCommit`** (resolved browsed directory or **`_currentFolderPath`**) so reconciliation uses the pinned exhaustion path. **`ExecuteBrowserTreeViewportAfterMutationCoreAsync`** scrolls the tree to offset zero only when there is no active browse directory; otherwise unpinned exhaustion enqueues a final Normal-priority bring-into-view attempt. When **`preferTreeSelectionBeforeBrowsedFolder`** is true (find-in-tree), the pass still falls back to the browsed-folder row after selection and pin attempts. |

Long-running wizard work **outside** `Refresh*` (e.g. bulk image recycle/permanent deletes inside **`WizardExecuteImageRecycleOrPermanentBatchAsync`**, directory merge/move, recycle of whole folder) also runs under **`EnterBrowserPaneMutation`** so navigation stays blocked until refresh (or **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`**) and **awaited** viewport scheduling complete. Bulk recycle uses a single shell **`IFileOperation`** when possible; permanent deletes after explicit approval may run with limited parallelism (`File.Delete`), all on the UI STA thread except **`Task.Run`** workers for deletes.

## Races addressed

1. **Overlay dismissed while work continues** — Escape/Close could hide the wizard while `async` work still ran; `IsDeleteArchiveWizardOverlayOpen` no longer blocked `nav.*`. Mutation depth blocks browse commands and async browse entry points until teardown finishes.
2. **Overlapping `BrowseNavigateByStepAsync` instances** — Still possible when depth is zero; with depth, new navigations are suppressed during mutation.
3. **Viewport vs navigation** — `RefreshBrowserPaneAfterWizard*` awaits **`ScheduleBrowserTreeViewportAfterMutationAsync`** before clearing nested mutation depth so the tree scroll pass finishes (within the implemented completion contract) before navigation unblocks. **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`** uses the same awaited viewport pass after a single populate + commit when the browsed folder was removed.
4. **Viewport vs deferred folder resort** — Without a pre-viewport flush, the awaited scroll could run while sibling folder rows were still in pre-resort order, then a timer-driven `FlushCoalescedFolderResorts` would reorder and rely on a second fire-and-forget viewport. Wizard `Refresh*` applies pending deferred resorts first so the awaited pass aligns the browsed folder after **final** order.
5. **Staged browse populate vs wizard tree refresh** — `AppendBrowserStagedToTargetAsync` used to append all image `TreeViewNode`s before the first `_populateBrowserGeneration` check, then insert folder chunks across `await Task.Yield()` boundaries while wizard `Refresh*` could reconcile the same sibling list under the same generation token. **`RefreshBrowserPaneAfterWizard*`** now increments **`_populateBrowserGeneration`** at entry, and staged append validates the token **before** appending image rows so superseded work cannot leave partial lists.
6. **Cold boot restored scroll vs deferred folder resort** — Startup metrics can still coalesce sibling resorts after the first browse populate; **`FlushCoalescedFolderResorts`** would schedule **`ScheduleBrowserTreeViewportAfterMutation`**, which pins selection or an unrealized browsed-root row and overrides **`ScrollViewer`** offsets restored from **`paths.browserTree`**. Cold boot sets **`_suppressBrowserTreeViewportMutationForColdBoot`** until **`ReleaseColdBootBrowserTreeViewportSuppressDeferred`** (two Low ticks after **`RestoreStartupBrowseAsync`** completes) so the first post-restore viewport pass does not fight **`ApplyColdBootBrowserTreeRestoreAsync`**.

## Cold boot manual QA (browser tree)

After changing cold-boot restore, verify on a full app restart (process exit, not just navigate away):

1. **Deep last image** — Last session had **`LastSelectedImage`** under a collapsed subfolder; after restart, the same file is selected, preview matches, and **`LastSelectedImage`** in settings is unchanged after a short idle (debounced persist).
2. **Expanded folders + scroll** — Several folders expanded and tree scrolled away from the top; restart restores expansion set and vertical offset (no repeated viewport “bounce” near the top).
3. **Aggregate folder sort** — Folder list sort **Total size** or **Image file count** so metrics resort runs; cold boot still lands on restored scroll without a long pin-to-top fight.
4. **Invalid / stale `browserTree` snapshot** — e.g. browse root moved on disk so **`IsRestoreRootMatching`** fails; tree still scrolls to origin and last image selection sync runs when the file still exists under the browse root.
