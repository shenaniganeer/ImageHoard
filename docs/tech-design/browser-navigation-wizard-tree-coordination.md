# Browser navigation, delete/archive wizard, and folder tree coordination

Agents and contributors: follow repository-wide guidance in [`AGENTS.md`](../../AGENTS.md) (source-of-truth order, PRD traceability, and closing `ImageHoard.App` before `dotnet build` / `dotnet test`).

This note describes how **image navigation**, **delete/move (wizard) processing**, and the **browser `TreeView`** interact. The subsystem spans several `MainWindow` partials; behavior is sensitive to **concurrent async** work.

## Maintenance rule

Any pull request that changes **keyboard/pointer dispatch for browse navigation**, **delete/archive wizard destructive flows**, **browser tree Find overlay**, **browser tree population, selection, or viewport scrolling**, or **preview enqueue/drain** in this area **must update this document** in the same PR (diagrams, flags, method names, or new entry points). Prefer linking to API symbols over pasting large code blocks.

## Browser tree selection and in-tree delete

- **`FolderTree`** uses default **single** `TreeView` selection (no `SelectionMode="Multiple"`). Pointer selection is handled by the control; **`FolderTree_SelectionChanged`** drives preview via **`ApplyBrowserTreeLeadPreviewAsync`** (image rows → **`OnFolderTreeImageRowSelectedAsync`**; otherwise **`ClearBrowserPaneImagePreviewStatePreserveTreeSelection`** when a preview was showing).
- **Context menu Delete** (`BrowserContextDelete_Click`, `MainWindow.BrowserTreeDelete.cs`): syncs selection to **`_browserContextMenuTargetNode`** when present, then **`ExecuteBrowserTreeDeleteForCurrentSelectionAsync`** (at most one nav row from **`FolderTree.SelectedNode`** via **`GetSelectedBrowserTreeNavNodes`**). After deletes, **`ClearBrowserTreeSelectionAfterDeleteAsync`** clears **`FolderTree.SelectedNode`** and refreshes preview.
- **Recycle Bin** deletes use **`WizardExecuteImageRecycleOrPermanentBatchAsync`** (files) and **`ExecuteSendFolderToRecycleBinAfterConfirmAsync`** (folders) after **`ContentDialog`** confirmation; selected paths are filtered and ordered with **`BrowserTreeDeletePathDedupe`** (`ImageHoard.Core.Browse`).

While the delete/archive wizard is visible, **Escape** and **Close** are limited when `IsBrowserPaneMutationInProgress` is true: the overlay root handler and `TryDismissTopModalForEscape` consume Escape without dismissing; Close is disabled (`DeleteArchiveWizardPanel.SetBrowserPaneMutationBlocking`).

The overlay root handles **Left**/**Right** when focus is not in a text field so arrows still advance matches from buttons and options. **`IsDefault` on Search** is not used in XAML (WinUI markup compiler rejects `IsDefault` on this `UserControl` button); Enter still runs search from the query field. |

## Navigation pipeline (non-slideshow)

1. **`BrowseNavigateByStep`** (`MainWindow.BrowserPane.cs`) increments the nav counter and starts **`BrowseNavigateByStepAsync`** without awaiting (fire-and-forget).
2. **`BrowseNavigateByStepAsync`** resolves browse context from `_browseNavAnchorPath`, tree selection, `_currentImageFullPath`, `_currentFolderPath`, loads paths, computes index, then **`EnqueuePreviewNavigation`** and tree sync.
3. **`BrowseNavigateSiblingFolderFromInput`** → **`BrowseNavigateSiblingFolderAsync`** for `nav.nextDirectory` / `nav.prevDirectory` (and for **`browse.treeNext` / `browse.treePrevious`** when a folder row is selected). Optional **`contextDirectoryOverride`** skips **`BrowseContextDirectory.Resolve`** after validating the path is under the browse root (**`BrowseContextImageSequence.IsContextDirectoryUnderBrowseRoot`**). Uses **`_populateBrowserGeneration`** to drop stale results after repopulation; wizard **`RefreshBrowserPaneAfterWizardImageDeletesAsync` / `RefreshBrowserPaneAfterWizardUndoAsync`** and **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`** (via **`NavigateToFolderAsync`**) also advance this generation so staged folder population cannot interleave after tree reconciliation.

When `IsBrowserPaneMutationInProgress` is true, **`BrowseNavigateByStep`** returns immediately (no async work); **`BrowseNavigateByStepAsync`** and **`BrowseNavigateSiblingFolderAsync`** no-op at entry.

## Preview drain

`MainWindow.PreviewNavigation.cs`: **`EnqueuePreviewNavigation`**, **`DrainPreviewNavigationCoreAsync`**, coalescing, **`SyncTreeSelectionToImagePath`**. Slideshow arrow keys use **`TryHandleSlideshowKeys`** (`MainWindow.P0.cs`), which returns false (no navigation) when `IsBrowserPaneMutationInProgress` is true.

### Sort-flag target path vs preview (`TryGetSortFlagTargetPath`)

`MainWindow.BrowserPane.cs`: **`TryGetSortFlagTargetPath`** chooses the file that **Keep / Delete / Unset** commands (`sort.flag*` and legacy K/D/U) affect. The info overlay’s flag glyph is driven by **`_currentImageFullPath`**, so if the tree’s selected **`ImageRow`** lags the committed preview (e.g. slideshow navigation without a matching selection update), commands must still target the **preview** path or the UI looks broken in fullscreen.

Resolution order:

1. While **`_slideshowUiActive`** and the preview path is non-empty → use **`_currentImageFullPath`**.
2. Else if the tree has an **`ImageRow`** selected and it **differs** from **`_currentImageFullPath`** (both set) → use **`_currentImageFullPath`** (preview wins over stale selection).
3. Else if an **`ImageRow`** is selected → use that row’s path (including “selection with no preview” cases).
4. Else → **`_currentImageFullPath`** when set.

After a successful bitmap commit in **`DecodeAndCommitPreviewAsync`**, when **`_slideshowUiActive`**, the app calls **`SyncTreeSelectionToImagePath(path)`** (suppresses selection-changed preview enqueue so **`StopSlideshowSession`** does not run) so the tree anchor stays aligned with the slide for other features.

## Destructive / reconciliation pipeline (wizard + browser)

| Step | API | Role |
|------|-----|------|
| User confirms delete/move | `WizardExecute*` in `MainWindow.DeleteArchiveWizard.cs` | File system and navigation side effects. **Move to archive** builds a case-insensitive union of (optional) inverse-keep paths and all **Delete**-flagged image paths in the folder, then runs the same recycle/permanent batch as other wizard deletes before `MergeMoveDirectoryAsync`. When that batch is immediately followed by a directory merge, the batch may pass **`deferBrowserPaneRefresh: true`** so the UI reconciles once after the merge succeeds (or once on merge failure using captured delete stats). |
| Reconcile tree + preview | `RefreshBrowserPaneAfterWizardImageDeletesAsync`, `RefreshBrowserPaneAfterWizardUndoAsync` (`MainWindow.BrowserPane.cs`) | Syncs image rows with disk, updates aggregates, refocuses selection. Wrapped with **`EnterBrowserPaneMutation` / `LeaveBrowserPaneMutation`** (try/finally). Each method **`Interlocked.Increment`s `_populateBrowserGeneration`** at the start of the refresh so in-flight **`AppendBrowserStagedToTargetAsync`** work from the prior browse cannot resume with a stale token after disk sync. |
| Wizard parent navigation (single bump + viewport) | **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`** (`MainWindow.BrowserPane.cs`) | After **folder recycle** or **Move to archive** when the removed folder was the browsed folder: calls **`NavigateToFolderAsync(parent, suppressViewportAfterRootPopulate: true)`** (one generation increment and root repopulate **without** populate-time viewport scheduling), then **`ClearImageSelectionAndPreviewCore`**, **`CommitIncrementalFolderPreviewAndSelectionAsync`**, **`PrepareBrowserTreeViewportAfterWizardMutation`**, and **one** awaited **`ScheduleBrowserTreeViewportAfterMutationAsync(GetBrowserTreeViewportPinPathAfterBrowseCommit())`**. Avoids the prior **Navigate + `RefreshBrowserPaneAfterWizardImageDeletesAsync`** double increment and double structural sync on **`FolderTree.RootNodes`**. |
| Defer batch refresh for chained IO | **`WizardExecuteImageRecycleOrPermanentBatchAsync(..., deferBrowserPaneRefresh: true)`** (`MainWindow.DeleteArchiveWizard.cs`) | Skips end-of-batch **`RefreshBrowserPaneAfterWizardImageDeletesAsync`**; stores succeeded delete stats and refocus context in **`_deferredWizardBatchSucceededStats`** / **`_deferredWizardBatchRefocusContext`**. **`ClearDeferredWizardBatchBrowserRefreshCapture`** runs after a successful **`MergeMoveDirectoryAsync`**; on merge failure, **`RefreshBrowserPaneAfterWizardImageDeletesAsync`** runs once with the deferred payload. |
| Suppress viewport on root populate | **`NavigateToFolderAsync(path, suppressViewportAfterRootPopulate)`**, **`PopulateBrowserRootsCoreAsync`**, **`FinalizeBrowserRootPopulateAfterImmediateAppend`**, **`AppendBrowserStagedToTargetAsync`** (`MainWindow.BrowserPane.cs`) | When true, root browse populate skips **`ScheduleBrowserTreeViewportAfterMutation`** so a caller (e.g. **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`**) performs a single scroll pass after selection commit. |
| Disk ↔ tree image merge | `SynchronizeFolderImageRowsWithDiskAsync` (`MainWindow.BrowserPane.cs`) | Dedupes sorted disk image rows by full path (case-insensitive) before merge; detects duplicate image paths or duplicate `TreeViewNode` references among sibling image rows (and duplicate refs inside the merged disk-order list) to skip the structural no-op fast-path and rebuild; merge always appends one `TreeViewNode` per deduped row. After wizard **`Refresh*`**, the browsed-folder sync uses **`forceStructuralImageRebuild: true`** via **`SynchronizeCurrentFolderImageNodesWithDiskAsync`** so the **`identicalStructure`** no-op path is skipped for **`FolderTree.RootNodes`**. |
| Tree expand (user) | `FolderTree_OnExpanding` (`MainWindow.BrowserPane.cs`) | When the user expands a folder row: if children were already materialized, **`SynchronizeFolderImageRowsWithDiskAsync`** re-lists that folder’s images vs disk (unless **`IsBrowserPaneMutationInProgress`** or inline rename **`_renameTargetNode`** is set); **`EnqueueFolderMetricsScanIfNeeded`** runs for the folder subtree. First-time expand still uses **`PopulateFolderTreeNodeChildrenAsync`**. |
| Browse nav from tree | `GetBrowseNavigationPathsForContextAsync`, `DedupePathsPreserveOrder`, `CollectImageNodesForBrowseContextDirectory` (`MainWindow.BrowserPane.cs`) | Tree-backed path lists dedupe case-insensitively with first occurrence kept so overlay `1/N` and sequential nav match distinct files; image node collection uses first materialized row per path. |
| Wizard refresh sibling sync | `RefreshBrowserPaneAfterWizardImageDeletesAsync`, `RefreshBrowserPaneAfterWizardUndoAsync` (`MainWindow.BrowserPane.cs`) | Tracks which sibling `IList<TreeViewNode>` instances were already reconciled (`RootNodes` plus expanded subfolders) and skips a second `SynchronizeFolderImageRowsWithDiskAsync` on the same list reference in one refresh pass. |
| Pick next image or folder row | `CommitIncrementalFolderPreviewAndSelectionAsync` | Uses `BrowserTreeRefocusAfterWizardContext` when supplied. |
| Deferred resort flush (wizard) | `PrepareBrowserTreeViewportAfterWizardMutation` (`MainWindow.BrowserPane.cs`) | When folder list sort is **aggregate size** or **image file count**, `RequestCoalescedFolderResortForTouchedFolderPaths` debounces sibling resorts. Wizard `Refresh*` stops that timer and applies any **pending** coalesced sibling sorts **after** commit and **before** the awaited viewport so the tree order matches sort keys when pinning. No-op for name / date sorts (no deferred pending set). |
| Viewport | `ScheduleBrowserTreeViewportAfterMutation` (fire-and-forget) and **`ScheduleBrowserTreeViewportAfterMutationAsync`** (awaitable) | Pins the browsed folder row (or selection) into view after mutations; retries on `DispatcherQueue` when containers are not realized. **`TryBringFolderTreeNodeToTop`** (`MainWindow.BrowserPane.cs`) walks the `FolderTree` visual tree for the inner **`TreeViewList`** and calls **`ScrollIntoView(node, Leading)`** before **`TreeViewItem.StartBringIntoView`**, because BringIntoView alone is unreliable for off-screen virtualized rows. Wizard **`RefreshBrowserPaneAfterWizardImageDeletesAsync`** / **`RefreshBrowserPaneAfterWizardUndoAsync`** pass a pin path from **`GetBrowserTreeViewportPinPathAfterBrowseCommit`** (resolved browsed directory or **`_currentFolderPath`**) so reconciliation uses the pinned exhaustion path. **`ExecuteBrowserTreeViewportAfterMutationCoreAsync`** scrolls the tree to offset zero only when there is no active browse directory; otherwise unpinned exhaustion enqueues a final Normal-priority bring-into-view attempt. When **`preferTreeSelectionBeforeBrowsedFolder`** is true (find-in-tree), the pass still falls back to the browsed-folder row after selection and pin attempts. |

Long-running wizard work **outside** `Refresh*` (e.g. per-file recycle loop, directory merge/move, recycle of whole folder) also runs under **`EnterBrowserPaneMutation`** so navigation stays blocked until refresh (or **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`**) and **awaited** viewport scheduling complete.

## Races addressed

1. **Overlay dismissed while work continues** — Escape/Close could hide the wizard while `async` work still ran; `IsDeleteArchiveWizardOverlayOpen` no longer blocked `nav.*`. Mutation depth blocks browse commands and async browse entry points until teardown finishes.
2. **Overlapping `BrowseNavigateByStepAsync` instances** — Still possible when depth is zero; with depth, new navigations are suppressed during mutation.
3. **Viewport vs navigation** — `RefreshBrowserPaneAfterWizard*` awaits **`ScheduleBrowserTreeViewportAfterMutationAsync`** before clearing nested mutation depth so the tree scroll pass finishes (within the implemented completion contract) before navigation unblocks. **`ReconcileBrowserPaneAfterWizardNavigateToParentAsync`** uses the same awaited viewport pass after a single populate + commit when the browsed folder was removed.
4. **Viewport vs deferred folder resort** — Without a pre-viewport flush, the awaited scroll could run while sibling folder rows were still in pre-resort order, then a timer-driven `FlushCoalescedFolderResorts` would reorder and rely on a second fire-and-forget viewport. Wizard `Refresh*` applies pending deferred resorts first so the awaited pass aligns the browsed folder after **final** order.
5. **Staged browse populate vs wizard tree refresh** — `AppendBrowserStagedToTargetAsync` used to append all image `TreeViewNode`s before the first `_populateBrowserGeneration` check, then insert folder chunks across `await Task.Yield()` boundaries while wizard `Refresh*` could reconcile the same sibling list under the same generation token. **`RefreshBrowserPaneAfterWizard*`** now increments **`_populateBrowserGeneration`** at entry, and staged append validates the token **before** appending image rows so superseded work cannot leave partial lists.
